\section*{Questão 1: Auditoria de Lotes (Conceito)}
Em uma fábrica de peças automotivas, os inspetores de qualidade registram a quantidade de peças defeituosas encontradas diariamente. Esses registros são armazenados em um vetor (array) de inteiros. Para automatizar o relatório semanal, você deve implementar um sistema que calcule o total de peças defeituosas utilizando \textbf{recursividade linear}.

\textbf{Requisitos:}
\begin{itemize}
    \item Implemente um método recursivo chamado \texttt{somarDefeitos} que receba o array de registros e o tamanho do array (ou o índice atual) como parâmetros.
    \item O método deve retornar a soma total dos elementos do array.
    \item Identifique e implemente claramente o \textbf{caso base} e o \textbf{passo recursivo}.
    \item Crie uma classe \texttt{Main} para testar o seu método com o array \texttt{\{2, 5, 1, 0, 4\}}.
\end{itemize}

\textbf{Pergunta Reflexiva:} Explique, com suas palavras, o que acontece com a pilha de execução (\textit{Call Stack}) do Java a cada nova chamada recursiva do método \texttt{somarDefeitos}. Como as variáveis locais e o estado de cada chamada são mantidos em memória até que o caso base seja atingido?

\section*{Questão 2: Validador de Códigos de Segurança (Prática)}
Um sistema de cofre digital utiliza códigos de acesso simétricos para liberar a abertura das portas. Um código é considerado válido apenas se for um palíndromo perfeito (ou seja, lido da mesma forma da esquerda para a direita e da direita para a esquerda, ignorando espaços e diferenças entre maiúsculas e minúsculas).

\textbf{Requisitos:}
\begin{itemize}
    \item Desenvolva um método recursivo \texttt{isPalindromo} que receba uma \texttt{String} (o código) e os índices de início e fim da análise.
    \item O método deve retornar um valor \texttt{boolean} indicando se o código é válido.
    \item Realize o tratamento para ignorar espaços em branco e letras maiúsculas/minúsculas da \texttt{String} original na classe principal antes de passar a informação para o método recursivo.
    \item A recursão deve comparar os caracteres das extremidades e se aproximar do centro a cada chamada (recursão de cauda/linear).
    \item Teste sua implementação com os códigos \texttt{"Ovo"}, \texttt{"Ame o poema"} e \texttt{"Senha123"}.
\end{itemize}

\textbf{Pergunta Reflexiva:} Em implementações recursivas como esta, a definição correta do caso base é vital. O que ocorreria nos bastidores (em termos de consumo de recursos e comportamento da JVM) se o caso base nunca fosse alcançado? Justifique.

\section*{Questão 3: Rastreador de Transações Financeiras (Desafio)}
Um banco processa milhões de transações diárias. Para otimizar o sistema de auditoria e detecção de fraudes, os IDs das transações (números inteiros) são armazenados em um array estritamente ordenado de forma crescente. Sua missão é implementar um motor de busca de alto desempenho para localizar rapidamente o índice de uma transação suspeita, seguindo o paradigma de divisão e conquista.

\textbf{Requisitos:}
\begin{itemize}
    \item Implemente um método recursivo de \textbf{Busca Binária} chamado \texttt{buscarTransacao} que receba o array de transações, o ID desejado e os limites do sub-array (início e fim).
    \item Se a transação for encontrada, retorne o índice correspondente no array.
    \item Se a transação não existir, retorne \texttt{-1}.
    \item Demonstre o funcionamento na classe \texttt{Main} utilizando um array contendo pelo menos 10 IDs ordenados e realize buscas por um ID existente e por um inexistente.
\end{itemize}

\textbf{Pergunta Reflexiva:} Ao utilizar a estratégia de divisão e conquista da Busca Binária recursiva, o tamanho do problema cai pela metade a cada chamada. Como essa característica afeta a profundidade máxima da pilha de chamadas (\textit{Call Stack}) em comparação com uma busca recursiva simples de ponta a ponta (busca linear) para o mesmo array? Explique a vantagem de desempenho com base nisso.

\section*{Questão 4: Controle de Vagas de Estacionamento (Conceito)}
Uma empresa de mobilidade urbana opera um estacionamento inteligente com um número fixo de vagas numeradas (de 0 a $N-1$). O sistema mantém um \textbf{array primitivo de Strings} de tamanho fixo, onde cada posição armazena a placa do veículo estacionado ou \texttt{null} se a vaga estiver livre. Você deve implementar um painel de controle básico para esse estacionamento.

\textbf{Requisitos:}
\begin{itemize}
    \item Declare um array de \texttt{String} com capacidade fixa de \textbf{8 vagas}.
    \item Implemente um método \texttt{estacionar(String[] vagas, int posicao, String placa)} que registre um veículo em uma posição específica. Se a posição já estiver ocupada, o método deve exibir um aviso e \textbf{não} sobrescrever a placa existente.
    \item Implemente um método \texttt{liberar(String[] vagas, int posicao)} que remova o veículo da posição informada (atribuindo \texttt{null}).
    \item Implemente um método \texttt{exibirPainel(String[] vagas)} que percorra todo o array e imprima o estado de cada vaga (livre ou ocupada com a respectiva placa).
    \item Na classe \texttt{Main}, simule a seguinte sequência: estacione 4 veículos em posições à sua escolha, tente estacionar um 5º veículo em uma posição já ocupada, libere uma vaga e exiba o painel final.
\end{itemize}

\textbf{Pergunta Reflexiva:} O array de vagas foi criado com tamanho fixo de 8 posições. Suponha que o estacionamento precise expandir para 12 vagas. Explique, do ponto de vista da alocação de memória contígua, por que \textbf{não é possível} simplesmente ``aumentar'' o array existente. Qual seria a estratégia necessária para acomodar as novas vagas e qual o custo computacional envolvido nessa operação?

\section*{Questão 5: Gerenciador de Playlist Musical (Prática)}
Uma plataforma de streaming de música permite que o usuário crie playlists com um limite máximo de músicas. Internamente, a playlist é representada por um \textbf{array primitivo de Strings} de capacidade fixa e uma variável \texttt{tamanho} que controla quantas músicas foram efetivamente inseridas. Você deve construir um gerenciador completo que demonstre as operações de inserção e remoção e seus custos.

\textbf{Requisitos:}
\begin{itemize}
    \item Crie uma classe \texttt{Playlist} que encapsule:
    \begin{itemize}
        \item Um array \texttt{String[] musicas} com capacidade fixa de \textbf{7}.
        \item Um atributo \texttt{int tamanho} (inicialmente 0) para rastrear a quantidade real de músicas.
    \end{itemize}
    \item Implemente o método \texttt{adicionarNoFim(String musica)} — insere no final da playlist. Caso a playlist esteja cheia, exiba uma mensagem de erro.
    \item Implemente o método \texttt{inserirNaPosicao(int posicao, String musica)} — insere uma música em uma posição específica, \textbf{deslocando (shift)} todos os elementos subsequentes uma posição para a direita. Valide se a posição é válida e se há espaço disponível.
    \item Implemente o método \texttt{removerDaPosicao(int posicao)} — remove a música na posição indicada, \textbf{deslocando (shift)} todos os elementos subsequentes uma posição para a esquerda para preencher o ``buraco''. Atribua \texttt{null} à última posição antes ocupada e decremente \texttt{tamanho}.
    \item Implemente o método \texttt{exibir()} que mostre apenas as músicas válidas (do índice 0 até \texttt{tamanho - 1}).
    \item Na classe \texttt{Main}, demonstre: adicione 5 músicas ao final, insira uma música na posição 2, remova a música na posição 0, e exiba a playlist após cada operação. Imprima \textbf{quantos deslocamentos (shifts)} foram necessários em cada inserção e remoção.
\end{itemize}

\textbf{Pergunta Reflexiva:} Ao inserir uma música na posição 0 de uma playlist que já contém $N$ músicas, todos os $N$ elementos precisam ser deslocados uma posição para a direita. Qual é a complexidade de tempo (Big-O) dessa operação no \textbf{pior caso}? Compare com a inserção no final. Por que essa diferença é uma limitação fundamental dos arrays primitivos?

\section*{Questão 6: Sistema de Senhas de Atendimento (Desafio)}
Um hospital utiliza um sistema de fila de atendimento baseado em senhas numéricas. Devido a uma restrição do sistema legado, a fila é implementada com um \textbf{array primitivo de inteiros} de tamanho fixo. O sistema precisa lidar com situações do mundo real como: pacientes que desistem (saem da fila), pacientes prioritários que precisam ser encaixados em posições específicas, e a possibilidade de a fila lotar. Além disso, o administrador quer uma opção de \textbf{redimensionamento manual} caso a capacidade se esgote.

\textbf{Requisitos:}
\begin{itemize}
    \item Crie uma classe \texttt{FilaDeSenhas} com:
    \begin{itemize}
        \item Um array \texttt{int[] senhas} com capacidade inicial de \textbf{5}.
        \item Um atributo \texttt{int quantidade} para controlar o número real de senhas na fila.
        \item Um atributo \texttt{int proximaSenha} (iniciando em 1) para gerar senhas sequenciais automaticamente.
    \end{itemize}
    \item Implemente \texttt{chamarProximaSenha()} — gera a próxima senha e a insere no \textbf{final} da fila. Se a fila estiver cheia, exiba ``Fila lotada!'' e \textbf{não} insira.
    \item Implemente \texttt{encaixarPrioritario(int posicao)} — gera uma nova senha e a insere na posição indicada, deslocando os demais. Valide se há espaço e se a posição é válida.
    \item Implemente \texttt{desistir(int posicao)} — remove a senha na posição informada, deslocando os elementos restantes para a esquerda.
    \item Implemente \texttt{redimensionar(int novaCapacidade)} — cria um \textbf{novo} array com a nova capacidade, copia todos os elementos existentes para ele e substitui o array interno. Trate o caso em que a nova capacidade é menor que a quantidade atual de elementos.
    \item Implemente \texttt{exibirFila()} para mostrar as senhas na ordem atual e a capacidade total.
    \item Na classe \texttt{Main}, simule o seguinte cenário:
    \begin{enumerate}
        \item Chame 5 senhas normais (lotando a fila).
        \item Tente chamar uma 6ª senha (deve mostrar ``Fila lotada!'').
        \item Redimensione a fila para capacidade 10.
        \item Encaixe um prioritário na posição 0.
        \item Faça a senha da posição 3 desistir.
        \item Exiba a fila final.
    \end{enumerate}
\end{itemize}

\textbf{Pergunta Reflexiva:} O método \texttt{redimensionar} cria um novo array e copia todos os elementos um a um. Sabendo que arrays ocupam \textbf{blocos contíguos de memória}, explique por que o Java (e a maioria das linguagens) não pode simplesmente ``esticar'' o array original na memória. O que acontece com o array antigo após a cópia? Relacione sua resposta com o conceito de \textit{Garbage Collection} da JVM.

\section*{Questão 7: Registro de Check-ins em Museu (Conceito)}
Um museu interativo recém-inaugurado quer digitalizar o registro de visitantes. A cada dia, um número completamente imprevisível de pessoas pode aparecer: desde 5 em uma segunda-feira chuvosa até 300 em um feriado nacional. Usar um array de tamanho fixo seria impraticável, pois o museu não sabe antecipadamente quantos visitantes receberá. Sua tarefa é utilizar um \texttt{ArrayList<String>} para criar um sistema flexível de check-in.

\textbf{Requisitos:}
\begin{itemize}
    \item Crie um \texttt{ArrayList<String>} chamado \texttt{visitantes}.
    \item Simule a chegada de \textbf{8 visitantes} ao museu, adicionando seus nomes à lista com o método \texttt{add()}.

    \item Exiba o \textbf{nome do 3º visitante} (índice 2) utilizando o método \texttt{get()}.
    \item Verifique se um visitante específico (à sua escolha) está na lista usando \texttt{contains()}.
    \item Exiba a lista completa de visitantes utilizando um laço \texttt{for-each}.

\textbf{Pergunta Reflexiva:} Um \texttt{ArrayList} internamente utiliza um array primitivo como estrutura de armazenamento. \section*{Questão 8: Gerenciador de Encomendas de Transportadora (Prática)}
Uma transportadora regional precisa de um sistema para gerenciar a lista de encomendas pendentes do dia. O volume de entregas varia drasticamente: em dias comuns são 20 pacotes, mas em datas promocionais (Black Friday, Natal) podem chegar mais de 500. O sistema deve permitir adicionar encomendas conforme chegam, consultar encomendas por posição, localizar uma encomenda pelo código de rastreio e remover encomendas que forem despachadas.

\begin{itemize}
    \item Crie uma classe \texttt{GerenciadorEncomendas} que encapsule:
    \begin{itemize}
        \item Um \texttt{ArrayList<String>} chamado \texttt{encomendas} para armazenar os códigos de rastreio.
    \end{itemize}
    \item Implemente o método \texttt{registrar(String codigoRastreio)} — adiciona uma nova encomenda ao final da lista usando \texttt{add()}. Antes de inserir, verifique com \texttt{contains()} se o código já existe (não permita duplicatas).
    \item Implemente o método \texttt{inserirUrgente(int posicao, String codigoRastreio)} — insere uma encomenda prioritária em uma posição específica usando \texttt{add(int index, E element)}. Valide o índice com tratamento de exceção (\texttt{IndexOutOfBoundsException}).

    \item Implemente o método \texttt{localizar(String codigoRastreio)} — retorna a posição (índice) da encomenda na fila usando \texttt{indexOf()}. Se não encontrada, retorne \texttt{-1}.
    \item Implemente o método \texttt{despachar(String codigoRastreio)} — remove a encomenda da lista usando \texttt{remove(Object)}. Exiba uma mensagem de sucesso ou de ``encomenda não encontrada''.
    \item Implemente o método \texttt{exibirPainel()} — lista todas as encomendas pendentes com seus índices e exiba a quantidade total via \texttt{size()}.
    \item Na classe \texttt{Main}, simule o seguinte cenário:

        \item Registre 6 encomendas com códigos fictícios (ex: ``BR123456789'').
        \item Tente registrar uma encomenda com código duplicado (deve ser rejeitada).
        \item Insira uma encomenda urgente na posição 0.
        \item Consulte a encomenda na posição 3.
        \item Localize o índice de uma encomenda pelo seu código.
        \item Despache 2 encomendas (removendo-as da lista).
        \item Exiba o painel final.
    \end{enumerate}
\end{itemize}

\textbf{Pergunta Reflexiva:} O método \texttt{add(int index, E element)} do \texttt{ArrayList} permite inserir em qualquer posição válida. Qual a vantagem real que o \texttt{ArrayList} oferece em relação ao array primitivo, considerando o redimensionamento?

\section*{Questão 9: Filtro de Alertas em Sistema de Monitoramento IoT (Desafio)}
Uma fábrica possui sensores IoT espalhados em diversas máquinas. Cada sensor gera alertas com um nível de severidade: \texttt{"INFO"}, \texttt{"WARNING"} ou \texttt{"CRITICAL"}. Ao final de cada turno, o operador precisa \textbf{filtrar} a lista de alertas removendo todos os alertas de nível \texttt{"INFO"} (considerados irrelevantes) para gerar um relatório apenas com os alertas que exigem atenção.

O grande desafio é: como remover elementos de um \texttt{ArrayList} \textbf{durante} uma iteração sem causar erros ou pular elementos?

\textbf{Requisitos:}
\begin{itemize}
    \item Crie uma classe \texttt{Alerta} com os atributos:
    \begin{itemize}
        \item \texttt{String sensor} — identificação do sensor (ex: ``Sensor-T01'').
        \item \texttt{String nivel} — nível de severidade (\texttt{"INFO"}, \texttt{"WARNING"} ou \texttt{"CRITICAL"}).
        \item \texttt{String mensagem} — descrição do alerta.
    \end{itemize}
    \item Crie uma classe \texttt{PainelDeAlertas} que encapsule um \texttt{ArrayList<Alerta>} e implemente:
    \begin{itemize}
        \item \texttt{registrar(Alerta alerta)} — adiciona um alerta à lista.
        \item \texttt{exibirTodos()} — exibe todos os alertas com seus dados.
    \end{itemize}
    \item Na classe \texttt{Main}, popule o painel com pelo menos \textbf{10 alertas} (misturando os três níveis de severidade).
    \item Implemente \textbf{três estratégias distintas de remoção} para remover os alertas \texttt{"INFO"}, cada uma em uma cópia separada da lista original:
    \begin{itemize}
        \item \textbf{Estratégia A (INCORRETA):} Utilize um \texttt{for} tradicional com índice (\texttt{for (int i = 0; i < lista.size(); i++)}) e \texttt{remove(i)} dentro do laço. Execute e observe o comportamento — elementos serão \textbf{pulados}.
        \item \textbf{Estratégia B (CORRETA):} Utilize um \texttt{Iterator<Alerta>} explícito com \texttt{iterator.hasNext()}, \texttt{iterator.next()} e \texttt{iterator.remove()} para remoção segura.
        \item \textbf{Estratégia C (MODERNA):} Utilize o método \texttt{removeIf()} com uma expressão lambda para remover os alertas de forma concisa e segura.
    \end{itemize}
    \item Após cada estratégia, exiba os alertas remanescentes e compare os resultados.
\end{itemize}

\textbf{Pergunta Reflexiva:} Na Estratégia A, ao remover o elemento no índice \texttt{i}, todos os elementos subsequentes são deslocados uma posição para a esquerda pelo \texttt{ArrayList}. Porém, a variável \texttt{i} é incrementada normalmente pelo \texttt{for}. Explique, passo a passo com um exemplo de 4 elementos, por que isso faz com que o elemento que estava imediatamente após o removido seja ``pulado'' pela iteração. Em seguida, explique como a abordagem de percorrer a lista \textbf{do final para o início} (\texttt{for (int i = lista.size()-1; i >= 0; i--)}) resolveria o problema sem usar \texttt{Iterator}. Por que essa técnica funciona?

\section*{Questão 10: Laboratório de Análises Clínicas (Conceito)}
Um laboratório de análises clínicas armazena amostras biológicas em tubos de ensaio. Cada tubo pode conter materiais de tipos diferentes: uma amostra de sangue é representada por uma \texttt{String} (o tipo sanguíneo, ex: ``O+''), uma contagem de leucócitos por um \texttt{Integer} e uma taxa de glicose por um \texttt{Double}. O sistema legado do laboratório utilizava um \textbf{array de \texttt{Object}} para guardar qualquer tipo de amostra, exigindo \textit{castings} explícitos e arriscando erros em tempo de execução (\texttt{ClassCastException}).

Sua missão é demonstrar o \textbf{antes e depois}: primeiro, reproduzir os problemas do código sem Generics; depois, criar uma classe genérica \texttt{Tubo<T>} que resolva esses problemas com segurança de tipos em tempo de compilação.

\textbf{Requisitos:}
\begin{itemize}
    \item \textbf{Parte A — Sem Generics (código problemático):}
    \begin{itemize}
        \item Crie um array \texttt{Object[] amostras} de tamanho 3.
        \item Armazene uma \texttt{String} (tipo sanguíneo), um \texttt{Integer} (leucócitos) e um \texttt{Double} (glicose) no array.
        \item Recupere o primeiro elemento e faça um \textit{casting} correto para \texttt{String}. Exiba o resultado.
        \item Agora, \textbf{propositalmente}, tente fazer um \textit{casting} do segundo elemento (\texttt{Integer}) para \texttt{String}. Capture a exceção \texttt{ClassCastException} com \texttt{try-catch} e exiba uma mensagem de erro explicativa.
    \end{itemize}
    \item \textbf{Parte B — Com Generics (solução segura):}
    \begin{itemize}
        \item Crie uma classe genérica \texttt{Tubo<T>} com:
        \begin{itemize}
            \item Um atributo privado \texttt{T amostra}.
            \item Um método \texttt{void inserir(T amostra)} para armazenar a amostra.
            \item Um método \texttt{T retirar()} que retorna a amostra armazenada.
        \end{itemize}
        \item Na classe \texttt{Main}, crie três tubos tipados: \texttt{Tubo<String>}, \texttt{Tubo<Integer>} e \texttt{Tubo<Double>}.
        \item Insira e retire amostras de cada tubo \textbf{sem} necessidade de \textit{casting}. Exiba os resultados.
        \item Demonstre que tentar inserir um tipo incompatível (ex: inserir um \texttt{Integer} em um \texttt{Tubo<String>}) gera um \textbf{erro de compilação}, e não de execução. Comente essa linha no código e explique com um comentário.
    \end{itemize}
\end{itemize}

\textbf{Pergunta Reflexiva:} Você observou que, com Generics, o compilador Java impede a inserção de tipos incompatíveis \textbf{antes mesmo} de o programa ser executado. Porém, nos bastidores, o Java aplica um mecanismo chamado \textbf{Type Erasure}: durante a compilação, todas as informações de tipo genérico são removidas, e \texttt{T} é substituído por \texttt{Object} no bytecode final. Sabendo disso, explique: se o bytecode compilado de \texttt{Tubo<String>} e \texttt{Tubo<Integer>} é \textbf{idêntico} (ambos usam \texttt{Object} internamente), \textit{como} o compilador consegue rejeitar o código incorreto antes da compilação? Qual é a vantagem prática dessa abordagem para o desenvolvedor?

\section*{Questão 11: Cofre Digital Parametrizado (Prática)}
Uma empresa de segurança patrimonial oferece cofres digitais inteligentes para seus clientes. Cada cofre pode guardar \textbf{um único tipo} de item valioso: alguns clientes guardam documentos (representados como \texttt{String}), outros guardam quantias em dinheiro (\texttt{Double}) e outros guardam códigos de acesso (\texttt{Integer}). O sistema deve ser parametrizado com Generics para que cada cofre tenha segurança de tipos: um cofre de documentos não pode acidentalmente receber uma quantia monetária.

\textbf{Requisitos:}
\begin{itemize}
    \item Crie uma classe genérica \texttt{Cofre<T>} que encapsule:
    \begin{itemize}
        \item Um \texttt{ArrayList<T>} privado chamado \texttt{itens} para armazenar os objetos guardados.
        \item Um atributo \texttt{String dono} que identifica o proprietário do cofre.
        \item Um atributo \texttt{int capacidadeMaxima} que define o limite de itens do cofre.
    \end{itemize}
    \item Implemente os seguintes métodos:
    \begin{itemize}
        \item \texttt{boolean guardar(T item)} — adiciona um item ao cofre. Retorna \texttt{false} se o cofre estiver cheio, exibindo uma mensagem de ``Cofre lotado!''. Não permita itens duplicados (\texttt{contains()}).
        \item \texttt{T recuperar(int posicao)} — retorna o item na posição informada. Trate \texttt{IndexOutOfBoundsException} e retorne \texttt{null} em caso de posição inválida.
        \item \texttt{boolean contem(T item)} — verifica se um item específico está no cofre usando \texttt{contains()}.
        \item \texttt{T remover(int posicao)} — remove e retorna o item na posição indicada. Trate exceções.
        \item \texttt{void listarConteudo()} — exibe todos os itens armazenados, o nome do dono e quantas posições ainda estão disponíveis.
    \end{itemize}
    \item Na classe \texttt{Main}, crie e demonstre o funcionamento de \textbf{três cofres} distintos:
    \begin{enumerate}
        \item \texttt{Cofre<String>} do cliente ``Alice'' (capacidade 5): guarde 4 documentos, tente guardar um duplicado (deve ser rejeitado), recupere o documento na posição 2 e liste o conteúdo.
        \item \texttt{Cofre<Double>} do cliente ``Bruno'' (capacidade 3): guarde 3 quantias, tente guardar uma 4ª (deve exibir ``Cofre lotado!''), e remova a quantia na posição 1.
        \item \texttt{Cofre<Integer>} do cliente ``Carla'' (capacidade 4): guarde 2 códigos de acesso, verifique se um código específico está no cofre usando \texttt{contem()}.
    \end{enumerate}
\end{itemize}

\textbf{Pergunta Reflexiva:} Ao compilar o seu programa, o Java aplica \textbf{Type Erasure} e substitui o parâmetro \texttt{T} da classe \texttt{Cofre<T>} por \texttt{Object}. Isso significa que, no bytecode final, \textbf{só existe uma única classe} \texttt{Cofre} — e não três versões separadas (\texttt{Cofre\_String}, \texttt{Cofre\_Double}, \texttt{Cofre\_Integer}) como seria em linguagens como C++ com templates. Explique por que essa decisão de design da linguagem Java (ter uma única classe compartilhada no bytecode) impede o uso de certas operações em tempo de execução, como \texttt{new T()} ou \texttt{instanceof T}. Dica: se \texttt{T} vira \texttt{Object} no bytecode, o que o programa ``saberia'' sobre \texttt{T} durante a execução?

\section*{Questão 12: Inventário de Loja Virtual com Limites de Tipo (Desafio)}
Uma loja virtual vende diferentes categorias de produtos: eletrônicos, livros e roupas. Todos os produtos compartilham atributos em comum (nome, preço), mas cada categoria possui dados específicos. O sistema de inventário precisa de uma estrutura genérica que funcione \textbf{apenas com subtipos de \texttt{Produto}} — rejeitando qualquer outro tipo (como \texttt{String} ou \texttt{Integer}) em tempo de compilação. Além disso, a equipe de relatórios precisa de um método utilitário que consiga calcular o valor total de \textbf{qualquer lista de produtos}, independentemente da categoria, usando \textbf{Wildcards}.

\textbf{Requisitos:}
\begin{itemize}
    \item Crie uma classe abstrata \texttt{Produto} com:
    \begin{itemize}
        \item Atributos protegidos: \texttt{String nome} e \texttt{double preco}.
        \item Construtor parametrizado e métodos \texttt{getNome()} e \texttt{getPreco()}.
        \item Um método abstrato \texttt{String descricaoDetalhada()} que cada subcategoria deve implementar.
        \item Sobrescreva \texttt{toString()} para exibir nome e preço formatado.
    \end{itemize}
    \item Crie \textbf{três subclasses concretas} de \texttt{Produto}:
    \begin{itemize}
        \item \texttt{Eletronico} — com atributo adicional \texttt{int garantiaMeses}.
        \item \texttt{Livro} — com atributo adicional \texttt{String autor}.
        \item \texttt{Roupa} — com atributo adicional \texttt{String tamanho}.
    \end{itemize}
    \item Crie uma classe genérica \texttt{Inventario<T extends Produto>} que encapsule:
    \begin{itemize}
        \item Um \texttt{ArrayList<T>} privado chamado \texttt{estoque}.
        \item \texttt{void adicionar(T produto)} — insere um produto no estoque.
        \item \texttt{T buscarPorNome(String nome)} — percorre a lista e retorna o primeiro produto cujo nome corresponda (ignore maiúsculas/minúsculas). Retorne \texttt{null} se não encontrado.
        \item \texttt{void listar()} — exibe todos os produtos com sua \texttt{descricaoDetalhada()}.
        \item \texttt{int quantidade()} — retorna o total de produtos no estoque.
    \end{itemize}
    \item Crie uma classe utilitária \texttt{RelatorioUtil} com um método \textbf{estático} que use \textbf{Wildcards}:
    \begin{itemize}
        \item \texttt{static double calcularValorTotal(ArrayList<? extends Produto> produtos)} — percorre qualquer lista de subtipos de \texttt{Produto} e retorna a soma dos preços.
    \end{itemize}
    \item Na classe \texttt{Main}, demonstre:
    \begin{enumerate}
        \item Crie um \texttt{Inventario<Eletronico>} e adicione 3 eletrônicos.
        \item Crie um \texttt{Inventario<Livro>} e adicione 3 livros.
        \item Crie um \texttt{Inventario<Roupa>} e adicione 2 roupas.
        \item Liste o conteúdo de cada inventário.
        \item Utilize \texttt{RelatorioUtil.calcularValorTotal()} para calcular e exibir o valor total de \textbf{cada} inventário separadamente — demonstrando que o método aceita \texttt{ArrayList<Eletronico>}, \texttt{ArrayList<Livro>} e \texttt{ArrayList<Roupa>} graças ao Wildcard.
        \item Demonstre (em um comentário explicativo) que tentar criar \texttt{Inventario<String>} causa erro de compilação por violar o limite \texttt{T extends Produto}.
    \end{enumerate}
\end{itemize}

\textbf{Pergunta Reflexiva:} O método \texttt{calcularValorTotal} aceita \texttt{ArrayList<? extends Produto>} em vez de \texttt{ArrayList<Produto>}. Explique por que \texttt{ArrayList<Eletronico>} \textbf{NÃO} é subtipo de \texttt{ArrayList<Produto>} em Java (conceito de \textbf{invariância} de tipos genéricos), e como o Wildcard \texttt{? extends} resolve esse problema. Além disso, sabendo que o Java aplica \textbf{Type Erasure}, o que acontece com \texttt{<T extends Produto>} durante a compilação: \texttt{T} é substituído por \texttt{Object} ou por \texttt{Produto}? Justifique.
